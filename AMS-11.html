Unidad 3: Análisis Estructurado

Introducción a las Herramientas del Análisis Estructurado

El análisis estructurado es una metodología tradicional que considera los datos y los procesos que los transforman como entidades separadas. Esta aproximación utiliza un conjunto específico de herramientas que permiten modelar de manera clara y precisa los requerimientos de un sistema.

En esta clase estudiaremos las cuatro herramientas fundamentales del análisis estructurado que constituyen el núcleo de esta metodología:

  • Diagrama de Contexto: Define los límites del sistema y su relación con el entorno
  • Diagramas de Flujo de Datos (DFD): Muestran cómo los datos fluyen a través del sistema
  • Diccionario de Datos: Especifica la composición y estructura de todos los datos
  • Especificaciones de Procesos: Detallan la lógica de los procesos primitivos
Nota Importante: Estas herramientas trabajan de manera coordinada y deben mantener consistencia entre sí. La trazabilidad entre artefactos es fundamental para garantizar la calidad del análisis.

Según Pressman, "el propósito de los diagramas de flujo de datos es proveer un puente semántico entre los usuarios y los desarrolladores de sistemas". Esta capacidad de comunicación es lo que hace tan valiosas estas herramientas en el proceso de análisis.

1. Diagrama de Contexto

Definición y Propósito

El diagrama de contexto, también conocido como DFD de Nivel 0, es la representación más alta del sistema en el análisis estructurado. Su función principal es definir claramente los límites del sistema y mostrar cómo interactúa con las entidades externas de su entorno.

Objetivo Principal: Establecer el alcance del sistema identificando qué está dentro y qué está fuera de los límites del sistema a desarrollar.

Elementos del Diagrama de Contexto

Componentes Visuales

1. El Sistema Central: Representado por un único círculo (proceso) que contiene todo el sistema que estamos analizando. Este círculo debe tener un nombre descriptivo que identifique claramente el sistema.

2. Entidades Externas: Son elementos del entorno que interactúan con el sistema pero que están fuera de nuestro control. Pueden ser:

  • Personas o roles (usuarios, clientes, administradores)
  • Otros sistemas de software
  • Organizaciones externas
  • Dispositivos de hardware

3. Flujos de Datos: Representan la información que se intercambia entre el sistema y las entidades externas. Cada flujo debe estar etiquetado con un nombre significativo que describa los datos que transporta.

Reglas de Construcción

Regla 1: Un diagrama de contexto debe contener exactamente un proceso
Regla 2: No debe mostrar almacenes de datos internos
Regla 3: Todos los flujos deben conectar el sistema central con entidades externas

Proceso de Identificación de Entidades Externas

Para identificar correctamente las entidades externas, debemos preguntarnos:

  1. ¿Quién proporciona datos al sistema?
  2. ¿Quién recibe información del sistema?
  3. ¿Qué otros sistemas intercambian información?
  4. ¿Qué eventos externos debe procesar el sistema?

Ejemplo Práctico: Sistema CasaSegura

Este diagrama de contexto del sistema CasaSegura muestra claramente los límites del sistema y las cinco entidades externas que interactúan con él. Cada flujo de datos está etiquetado descriptivamente.

Validación del Diagrama de Contexto

Para validar nuestro diagrama de contexto, debemos verificar:

  • ¿Están identificadas todas las fuentes de datos?
  • ¿Están identificados todos los destinos de información?
  • ¿Los nombres de flujos son descriptivos y únicos?
  • ¿Las entidades externas están realmente fuera de nuestro control?
  • ¿El sistema central tiene un nombre apropiado?
Tip de Análisis: El diagrama de contexto debe ser comprensible para los stakeholders no técnicos, ya que define el alcance y los límites del proyecto de manera visual y clara.

2. Diagramas de Flujo de Datos (DFD)

Fundamentos de los DFD

Los Diagramas de Flujo de Datos representan el corazón del análisis estructurado. Adoptan un punto de vista "entrada-proceso-salida" para modelar cómo los datos fluyen a través del sistema y cómo son transformados por los diferentes procesos.

Definición: Un DFD es una representación gráfica que muestra el flujo de datos a través de un sistema de información, los procesos que transforman estos datos, y los lugares donde se almacenan.

Reglas de Construcción de DFD

Regla de Balanceo: Las entradas y salidas de un proceso en un nivel deben coincidir con las entradas y salidas de su descomposición en el nivel inferior
Regla de Numeración: Los procesos se numeran jerárquicamente (1, 2, 3 en nivel 1; 1.1, 1.2, 1.3 para subprocesos del proceso 1)
Regla de Etiquetado: Todos los flujos, procesos y almacenes deben tener nombres descriptivos y únicos
Regla de Conectividad: No pueden existir flujos directos entre entidades externas sin pasar por un proceso

Ejemplo Detallado: DFD Nivel 1 del Sistema CasaSegura

Proceso de Creación de DFD

Pressman sugiere seguir estos lineamientos para crear DFD efectivos:

  1. Comenzar con el Nivel 0: El diagrama debe ilustrar el software como una sola burbuja
  2. Anotar entradas y salidas principales: Identificar claramente los flujos de datos críticos
  3. Aislar procesos candidatos: Usar análisis gramatical para identificar verbos (procesos) y sustantivos (datos)
  4. Etiquetar significativamente: Todas las flechas y burbujas deben tener nombres descriptivos
  5. Mantener continuidad del flujo: Los datos deben ser consistentes entre niveles
  6. Refinar una burbuja a la vez: Descomponer progresivamente cada proceso

Técnica del Análisis Gramatical

El análisis gramatical es una técnica fundamental para identificar elementos del DFD a partir de descripciones textuales:

Sustantivos y frases sustantivadas → Candidatos a objetos de datos, entidades externas, almacenes

Verbos y frases verbales → Candidatos a procesos

Adjetivos → Candidatos a atributos

Ejemplo de análisis gramatical aplicado a CasaSegura:

"La función de seguridad CasaSegura permite que el propietario configure el sistema de seguridad cuando se instala, vigila todos los sensores conectados al sistema de seguridad e interactúa con el propietario a través de internet, una PC o un panel de control."

De este análisis obtenemos:

  • Entidades: Propietario, Panel de control, Sensores
  • Procesos: Configurar, Vigilar, Interactuar
  • Datos: Configuración del sistema, Estado de sensores

Criterios de Descomposición

La descomposición de procesos debe continuar hasta alcanzar procesos primitivos que cumplan estos criterios:

  • Cohesión funcional: El proceso realiza una sola función bien definida
  • Implementabilidad: Puede ser implementado como un componente de programa
  • Testabilidad: Su función puede ser verificada independientemente
  • Comprensibilidad: Su propósito es claro para stakeholders y desarrolladores
Regla de Oro: Continúe refinando los DFD hasta que cada burbuja realice una función simple y tenga "un solo pensamiento" (alta cohesión).

3. Diccionario de Datos

Concepto y Propósito

El diccionario de datos es un listado organizado que contiene definiciones precisas y rigurosas de todos los datos pertinentes al sistema. Su objetivo es que el analista tenga un entendimiento completo de todos los datos que forman parte de los procesos del sistema.

Definición: El diccionario de datos describe el significado de los flujos y almacenes que se muestran en los DFD, especificando la composición de los paquetes de datos, sus valores y unidades relevantes.

Importancia del Diccionario de Datos

El diccionario de datos es fundamental porque:

  • Elimina la ambigüedad en la definición de datos
  • Establece un vocabulario común para el proyecto
  • Facilita la comunicación entre analistas, desarrolladores y usuarios
  • Sirve como base para el diseño de bases de datos
  • Asegura la consistencia en el uso de datos
  • Apoya la trazabilidad entre artefactos

Notación del Diccionario de Datos

El diccionario de datos utiliza una notación específica para definir la estructura y composición de los datos:

Símbolo Significado Descripción Ejemplo
= Está compuesto de Define la composición de un elemento de datos Nombre = Primer_nombre + Apellido
+ Y (concatenación) Une elementos que aparecen juntos Dirección = Calle + Número + Ciudad
( ) Opcional Elemento que puede estar presente o no Nombre = Primer_nombre + (Segundo_nombre) + Apellido
{ } Iteración Elemento que se repite cero o más veces Estudiantes = {Nombre + Legajo}
[ ] Selección Elegir uno de varios elementos alternativos Forma_pago = [Efectivo | Tarjeta | Cheque]
@ Identificador Campo clave o identificador único Empleado = @ID_empleado + Nombre + Cargo
| OR (separador) Separa opciones alternativas Estado = [Activo | Inactivo | Suspendido]
* * Comentarios Información adicional descriptiva Edad = Número *Entre 0 y 120*

Organización del Diccionario de Datos

Estructura del Diccionario de Datos

Proceso de Construcción

Para construir un diccionario de datos efectivo, siga estos pasos:

  1. Identificar elementos: Extraer todos los flujos y almacenes de los DFD
  2. Descomponer elementos complejos: Analizar la estructura interna de cada elemento
  3. Definir elementos primitivos: Especificar tipos de datos, rangos, formatos
  4. Establecer relaciones: Documentar dependencias entre elementos
  5. Validar consistencia: Verificar coherencia con los DFD
  6. Documentar restricciones: Incluir reglas de negocio y validaciones

Tipos de Elementos de Datos

Elementos Primitivos: No se descomponen más (ej: Número, Texto, Fecha)

Elementos Compuestos: Formados por otros elementos (ej: Dirección = Calle + Número + Ciudad)

Elementos Opcionales: Pueden estar presentes o ausentes

Elementos Repetitivos: Aparecen múltiples veces en una estructura

Validaciones y Restricciones

El diccionario debe incluir restricciones que aseguren la integridad de los datos:

Ejemplos de Restricciones: Sensor_ID = Número *Entero entre 1 y 999, único en el sistema* Email = Texto *Formato: usuario@dominio.com, máximo 100 caracteres* Timestamp = Fecha_Hora *Formato: YYYY-MM-DD HH:MM:SS* Temperatura = Número *Decimal entre -50.0 y 80.0 grados Celsius* Password = Texto *Mínimo 8 caracteres, debe incluir letras y números*
Principio de Consistencia: Todos los flujos de datos que aparecen en los DFD deben estar definidos en el diccionario de datos, y viceversa. Esta consistencia es crucial para la trazabilidad del análisis.

4. Especificaciones de Procesos

Definición y Propósito

Las especificaciones de procesos, también conocidas como minispec o PSPEC (Process Specification), describen la lógica de transformación de los procesos primitivos identificados en los DFD. Su propósito es definir exactamente qué hace cada proceso con los datos de entrada para producir los datos de salida.

Objetivo: Proporcionar una descripción precisa, no ambigua y verificable de la lógica interna de cada proceso primitivo del sistema.

¿Cuándo Crear Especificaciones de Procesos?

Se crean especificaciones para:

  • Procesos primitivos: Aquellos que no se descomponen más en el DFD
  • Procesos con lógica compleja: Que requieren algoritmos o reglas específicas
  • Procesos críticos: Fundamentales para el funcionamiento del sistema
  • Procesos con múltiples condiciones: Que requieren toma de decisiones

Técnicas de Especificación

1. Lenguaje Estructurado (Pseudocódigo)

Utiliza construcciones similares a los lenguajes de programación pero en lenguaje natural:

Ejemplo: Proceso "Validar Password" SI password_ingresado = password_maestro ENTONCES validación = verdadero mensaje = "Acceso autorizado" SINO PARA cada entrada en tabla_passwords_secundarios SI password_ingresado = entrada.password ENTONCES validación = verdadero mensaje = "Acceso temporal autorizado" SALIR DEL BUCLE FIN SI FIN PARA SI validación = falso ENTONCES mensaje = "Acceso denegado" incrementar_contador_intentos() SI contador_intentos > 3 ENTONCES activar_alarma_seguridad() FIN SI FIN SI FIN SI

Validación de Especificaciones

Para asegurar la calidad de las especificaciones de procesos:

  • Completitud: Cubren todos los casos posibles de entrada
  • Consistencia: No contradicen otras especificaciones
  • Verificabilidad: Pueden ser probadas y validadas
  • Trazabilidad: Conectan con elementos del DFD y diccionario
  • Claridad: Son comprensibles por desarrolladores y usuarios
Criterio de Calidad: Una buena especificación de proceso debe permitir que cualquier desarrollador implemente el proceso de manera consistente, produciendo los mismos resultados para las mismas entradas.

5. Integración y Balanceo de Herramientas

Principios de Trazabilidad

La efectividad del análisis estructurado depende de la correcta integración entre sus herramientas. Cada elemento debe ser trazable a través de todos los artefactos:

Trazabilidad Horizontal: Entre elementos del mismo nivel (procesos, flujos, almacenes)

Trazabilidad Vertical: Entre diferentes niveles de abstracción (contexto → DFD nivel 1 → nivel 2...)

Trazabilidad Cruzada: Entre diferentes tipos de artefactos (DFD ↔ Diccionario ↔ Especificaciones)

Reglas de Balanceo

Regla de Conservación: Los flujos que entran y salen de un proceso en un nivel deben ser los mismos que entran y salen de su descomposición en el nivel inferior
Regla de Denominación: Los nombres de flujos deben ser consistentes entre niveles y estar definidos en el diccionario de datos
Regla de Completitud: Todo elemento que aparece en un DFD debe estar definido en el diccionario de datos

Proceso de Validación Cruzada

Para asegurar la consistencia entre herramientas:

  1. Verificación DFD → Diccionario: Todos los flujos del DFD están definidos
  2. Verificación Diccionario → DFD: No hay definiciones sin uso en DFD
  3. Verificación Especificaciones → DFD: Cada proceso primitivo tiene su PSPEC
  4. Verificación Balanceo: Entradas/salidas consistentes entre niveles
  5. Validación Semántica: Los nombres y significados son coherentes
Metodología de Verificación: Use herramientas automatizadas cuando sea posible, pero siempre complete con revisión manual para verificar la coherencia semántica y la calidad del análisis.

Conclusiones y Próximos Pasos

En esta clase hemos estudiado las cuatro herramientas fundamentales del análisis estructurado. Estas herramientas trabajan de manera integrada para proporcionar una visión completa y consistente de los requerimientos del sistema:

  • Diagrama de Contexto: Establece los límites y el alcance del sistema
  • DFD Multinivel: Descompone funcionalmente el sistema mostrando flujos de datos
  • Diccionario de Datos: Define precisamente todos los elementos de información
  • Especificaciones de Procesos: Detallan la lógica de transformación de datos

Recordatorio Clave: La calidad del análisis estructurado depende de la consistencia y trazabilidad entre todas estas herramientas. Cada elemento debe poder ser seguido desde su aparición en un DFD hasta su definición en el diccionario y su implementación en las especificaciones.

En la próxima clase estudiaremos la Trazabilidad y Gestión de Cambios, donde profundizaremos en las técnicas para mantener la consistencia del modelo de análisis cuando los requerimientos evolucionan.

Preparación para la Próxima Clase: Revisar los conceptos de matrices de trazabilidad y procedimientos de control de cambios. Practicar la validación cruzada entre DFD y diccionario de datos usando el caso de estudio CasaSegura.
